### **Юніт-тестування: теорія та практика**

Юніт-тестування є фундаментальною практикою розробки програмного забезпечення, яка полягає у перевірці окремих компонентів або методів, що називаються «одиницями роботи». Основна мета — переконатися, що кожна ізольована частина коду працює саме так, як очікував автор.

#### **Переваги та обмеження юніт-тестування порівняно з інтеграційним**

**Юніт-тестування (Unit Testing):**
*   **Переваги:** Дозволяє розробникам писати більш простий, читабельний та ефективний код, особливо при використанні підходу TDD (розробка через тестування). Воно фокусується виключно на коді, який знаходиться під контролем розробника, і не вимагає доступу до зовнішньої інфраструктури, що робить такі тести надзвичайно швидкими.
*   **Обмеження:** Юніт-тести не перевіряють питання інфраструктури, такі як взаємодія з базами даних, файловими системами або мережевими ресурсами. Вони також не можуть гарантувати, що різні модулі будуть коректно працювати разом.

**Інтеграційне тестування (Integration Testing):**
*   **Переваги:** На відміну від юніт-тестів, воно перевіряє здатність двох або більше компонентів функціонувати разом. Такі тести охоплюють ширший спектр системи і часто включають перевірку інфраструктурних зв'язків.
*   **Обмеження:** Вони складніші в налаштуванні, працюють повільніше і зазвичай важче локалізувати конкретну помилку в логіці окремого методу, ніж у випадку з юніт-тестами.

#### **Приклад класу та тестів (C# з використанням xUnit та Shouldly)**

Нижче наведено приклад класу `BankAccount`, який керує балансом користувача.

```csharp
public class BankAccount
{
    public decimal Balance { get; private set; }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Сума має бути додатною");
        Balance += amount;
    }

    public void Withdraw(decimal amount)
    {
        if (amount > Balance) throw new InvalidOperationException("Недостатньо коштів");
        Balance -= amount;
    }
}
```

**Тести для методів (використовуючи xUnit та Shouldly для зрозумілих повідомлень про помилки):**

1.  **Метод `Deposit`:**
    *   *Успішний сценарій:* Перевірка, що баланс збільшується після внесення коштів.
    *   *Граничний випадок:* Перевірка, що внесення від'ємної суми викликає виняток.
2.  **Метод `Withdraw`:**
    *   *Успішний сценарій:* Перевірка, що баланс зменшується після зняття коштів.
    *   *Граничний випадок:* Спроба зняти більше коштів, ніж є на рахунку.

#### **Використання Mock-об’єктів**

**Mock-об'єкти** — це інструменти (наприклад, з бібліотеки **Moq**), які дозволяють імітувати поведінку залежностей.

*   **Коли доцільно використовувати:**
    *   Коли тестований метод залежить від **зовнішніх факторів**, таких як бази даних, API або складні інфраструктурні компоненти, які не повинні впливати на юніт-тест.
    *   Для швидкого створення «заглушок» (stubs) для залежностей, які не потребують детальної перевірки в межах даного тесту.
    *   Для **верифікації**, чи був викликаний певний метод залежного об'єкта з правильними параметрами.
    *   Для тестування асинхронних методів, де потрібно налаштувати повернення `Task` або `ValueTask`.

*   **Коли можна обійтися без них:**
    *   Якщо клас є **самодостатнім** і не має зовнішніх залежностей (наприклад, чиста математична логіка).
    *   Коли залежність є простим об'єктом даних (Value Object або DTO).
    *   Якщо можна використовувати реальний об'єкт без втрати швидкості та ізоляції тесту.

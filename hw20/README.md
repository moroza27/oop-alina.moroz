# Adapter / Facade / Proxy: умови застосування та відмінності

## 1. Вступ

Adapter, Facade та Proxy належать до структурних патернів проєктування. Вони регулюють взаємодію між класами та об’єктами, проте мають різні цілі застосування: інтеграція, спрощення та контроль доступу.

# 2. Adapter

## 2.1. Призначення

Патерн **Adapter** використовується для забезпечення сумісності між класами з різними інтерфейсами.

Він перетворює інтерфейс одного класу на інтерфейс, який очікує клієнт.

## 2.2. Реальні сценарії застосування

1. Інтеграція сторонніх бібліотек.
2. Використання застарілих (legacy) компонентів у новій системі.
3. Робота з API, формат якого відрізняється від внутрішніх стандартів проєкту.

## 2.3. Проблема, яку вирішує

Adapter дозволяє інтегрувати несумісні компоненти без зміни їх вихідного коду.

# 3. Facade

## 3.1. Призначення

Патерн **Facade** надає спрощений інтерфейс до складної підсистеми.

Його метою є зменшення складності взаємодії клієнта з великою кількістю взаємопов’язаних класів.

## 3.2. Реальні сценарії застосування

1. Спрощення доступу до складних бібліотек.
2. Створення єдиної точки входу до підсистеми.
3. Організація високорівневого API для клієнтів.

## 3.3. Проблема, яку вирішує

Facade зменшує залежність клієнта від внутрішньої структури системи та підвищує її зрозумілість.

# 4. Proxy

## 4.1. Призначення

Патерн **Proxy** забезпечує контроль доступу до іншого об’єкта.

Proxy створює замісник (представника), який має той самий інтерфейс, що і реальний об’єкт.

## 4.2. Реальні сценарії застосування

1. Ліниве створення ресурсомістких об’єктів.
2. Перевірка прав доступу.
3. Кешування результатів.
4. Організація доступу до віддалених об’єктів.

## 4.3. Проблема, яку вирішує

Proxy дозволяє додати додаткову логіку (контроль, перевірку, кешування) без зміни основного об’єкта.

# 5. Ключові відмінності між патернами

| Патерн  | Основна мета                                     |
| ------- | ------------------------------------------------ |
| Adapter | Забезпечення сумісності інтерфейсів              |
| Facade  | Спрощення складної системи                       |
| Proxy   | Контроль доступу або додавання додаткової логіки |

### Принципова різниця:

* **Adapter** змінює інтерфейс.
* **Facade** спрощує взаємодію.
* **Proxy** зберігає інтерфейс, але додає контроль.

## 6. Висновок

Adapter, Facade та Proxy вирішують різні архітектурні завдання у структурі програмного забезпечення.

Adapter забезпечує інтеграцію несумісних компонентів, Facade спрощує доступ до складних систем, а Proxy реалізує контроль або додаткову поведінку при збереженні початкового інтерфейсу.

Раціональний вибір патерна залежить від конкретної задачі та архітектурних вимог проєкту.
